# Where case data comes from

Every field the Action Inbox renders, classified by who should produce it in the
real system. Five sources:

- **SYSTEM** — integrations / systems of record (Okta, Iru MDM, NetSuite, HiBob…).
  Deterministic lookups; never generated by a model.
- **ENGINE** — computed by our backend (ids, timers, state machines, retrieval).
- **HUMAN** — verbatim human content; a model must never rewrite it.
- **AGENT** — the owning domain agent's LLM judgment (mid-tier model, e.g. Sonnet).
- **LIGHT** — cheap/fast LLM (e.g. Haiku): compression, labeling, phrasing over
  facts that already exist. Safe because the facts are supplied, not recalled.

## Case core

| Field | Source | Notes |
|---|---|---|
| `id`, `state`, `arrival`, `updated`, `updatedPct` | ENGINE | Sequence, state machine, clocks. |
| `timer`, `slaPct` | ENGINE | From the channel's SLA config. |
| `title` | AGENT | Names the specific action + subject ("Unlock Okta for Priya N. …"), not the ticket subject. |
| `summary` | AGENT | First-person proposal — the agent's own voice. |
| `confidence` | AGENT + ENGINE | Model self-estimate, then calibrated against verified-outcome history per playbook (the engine owns the correction curve). |
| `hold` | ENGINE + LIGHT | Which policy held it is engine fact; the sentence is templated or LIGHT-phrased. |
| `verb`, `verbKey` | ENGINE | From the recommended option's action class. |
| `path` | ENGINE | Routing breadcrumb — pure telemetry. |
| `dot` | ENGINE | Derived from state. |

## Signal section

| Field | Source | Notes |
|---|---|---|
| `quote.body` | HUMAN | The requester's words, verbatim from the channel. Never paraphrased. |
| `quote.name/via/initials` | SYSTEM | Channel identity. |
| `quote.route`, `quote.status` | ENGINE | Routing + case state. |
| `why[]` | ENGINE + LIGHT | The *triggers* are policy-engine facts (below threshold, identity write, device changed). LIGHT turns triggers into readable lines. Don't let a big model invent reasons. |
| `sources` / `sourcesDetail[]` | ENGINE | The tool-call log of what the agent actually read. A record, not prose. |

## Decision section

| Field | Source | Notes |
|---|---|---|
| `options[0]` (recommended) | AGENT | The proposal the agent already formed before holding — see "Where do suggestions come from" below. |
| `options[1..2]` (variant/alternative) | AGENT over ENGINE candidates | Candidate actions come from the playbook's known branches + the tool registry (what's actually executable); the agent selects and words them. Keeps options grounded in real capabilities. |
| `options[].keys` | ENGINE | Fixed keybinding scheme. |
| `via` (action strip) | ENGINE | From the case's connected channels/tools. |

## Context side panel

| Field | Source | Notes |
|---|---|---|
| `context.identity[]` | SYSTEM | Okta/HRIS lookups. The `source`/`tone` tags are the point — provenance is the feature. |
| `context.roster[]` | SYSTEM | Org graph + who touched the case. |
| `context.devices[]` | SYSTEM | Iru MDM inventory — the canonical "obviously deterministic" example. |
| `context.apps[]` | SYSTEM | App/tenant state per system. |
| `context.scope` | LIGHT | One-line compression of the deterministic scope facts ("Unlock account · require password reset · SLA 22m"). |
| `context.prior[]` | ENGINE + LIGHT | Retrieval (embedding/keyword over case history) finds them; LIGHT writes the one-line relevance meta. |

## Conversation & discussion panels

| Field | Source | Notes |
|---|---|---|
| `conversation[]` | HUMAN + AGENT | Verbatim transcript; agent turns were AGENT at the time they happened. Artifacts are SYSTEM attachments. |
| `discussion[]` | HUMAN + AGENT | Human comments verbatim; @-mentioned agent replies are AGENT with their `ctx.read` scope recorded by ENGINE. |

## Brain panel

| Field | Source | Notes |
|---|---|---|
| `brain.reads[]` | SYSTEM + LIGHT | Each read is a logged world-model query (fact); LIGHT may compress the `value` line. |
| `brain.playbook` (id, name, origin, success) | ENGINE | Playbook library metadata; success % from verified outcomes. |
| `brain.playbook.steps[]` | ENGINE | Execution engine state. Step *labels* were authored when the playbook was learned (AGENT, once), not per case. |
| `brain.route` (model, path, cost, tokens) | ENGINE | Router telemetry. `route.reason` can be LIGHT-phrased from the router's decision record. |
| `brain.boundary.sent/withheld` | ENGINE | **Security property — must be the literal record of what crossed the boundary. Never model-generated.** |
| `brain.boundary.provider/retention/ledger` | ENGINE | Contract + ledger facts. |
| `brain.writes[]` | ENGINE | What each decision trains — policy-engine templates keyed to option classes. |

## Event history

| Field | Source | Notes |
|---|---|---|
| `events[]` (actor, time, icon) | ENGINE | Execution/audit log. |
| `events[].text` | LIGHT | One-line phrasing of a structured log entry; safe for a small model since the entry supplies all facts. |

**Rule of thumb:** records, provenance, and anything security-adjacent are SYSTEM/ENGINE.
Judgment (what to do, how sure) is the owning AGENT. Language over existing facts
(labels, one-liners, event text, scope lines) is LIGHT — this is most of the volume
and where fast models earn their keep.

## Where do suggestions come from?

Today's stand-in: `src/workflows/investigate-signal.ts` — one workflow generates the
whole package from the raw signal. Right shape for testing prompts, wrong shape for
the product.

In the real system the **owning agent generates the options itself, at hold time**,
as part of one immutable handover packet:

1. A personal/domain agent works the request. It has the world-model reads, the
   playbook position, and the boundary record — it already *formed* an intended
   action before policy held it.
2. When a hold triggers, the agent assembles the **HITL packet**: signal digest +
   evidence (`why`, `sources`, `brain.reads`) + its intended action as `options[0]` +
   variants. A separate "suggestion agent" would have to re-derive or hallucinate
   exactly the context the owning agent already holds — and would break training
   attribution.
3. Variants aren't free-generated: the playbook's known branches and the tool
   registry supply the *candidate set* (what is actually executable); the agent
   selects and words the best variant and the cautious alternative. Phrasing polish
   can drop to a LIGHT model.
4. The packet is immutable at handover. The operator's decision writes back to the
   ledger against the same agent that proposed it — that's what makes the decision
   a usable training signal ("reinforces PB-112").

So: keep suggestions with the owning agent, ground the candidates in the playbook +
tool registry, and reserve separate models for phrasing, not deciding.
